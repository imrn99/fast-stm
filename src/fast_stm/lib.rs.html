1//! This library implements
2//! [software transactional memory](https://en.wikipedia.org/wiki/Software_transactional_memory),
3//! often abbreviated with STM.
4//!
5//! It is designed closely to haskells STM library. Read Simon Marlow's
6//! *Parallel and Concurrent Programming in Haskell*
7//! for more info. Especially the chapter about
8//! Performance is also important for using STM in rust.
9//!
10//! With locks the sequential composition of two
11//! two threadsafe actions is no longer threadsafe because
12//! other threads may interfer in between of these actions.
13//! Applying a third lock to protect both may lead to common sources of errors
14//! like deadlocks or race conditions.
15//!
16//! Unlike locks Software transactional memory is composable.
17//! It is typically implemented by writing all read and write
18//! operations in a log. When the action has finished and
19//! all the used `TVar`s are consistent, the writes are commited as
20//! a single atomic operation.
21//! Otherwise the computation repeats. This may lead to starvation,
22//! but avoids common sources of bugs.
23//!
24//! Panicing within STM does not poison the `TVar`s. STM ensures consistency by
25//! never committing on panic.
26//!
27//! # Usage
28//!
29//! You should only use the functions that are transaction-safe.
30//! Transaction-safe functions don't have side effects, except those provided by `TVar`.
31//! Mutexes and other blocking mechanisms are especially dangerous, because they can
32//! interfere with the internal locking scheme of the transaction and therefore
33//! cause deadlocks.
34//!
35//! Note, that Transaction-safety does *not* mean safety in the rust sense, but is a
36//! subset of allowed behavior. Even if code is not transaction-safe, no segmentation
37//! faults will happen.
38//!
39//! You can run the top-level atomic operation by calling `atomically`.
40//!
41//!
42//! ```
43//! # use fast_stm::atomically;
44//! atomically(|trans| {
45//!     // some action
46//!     // return value as `Result`, for example
47//!     Ok(42)
48//! });
49//! ```
50//!
51//! Nested calls to `atomically` are not allowed. A run-time check prevents this.
52//! Instead of using atomically internally, add a `&mut Transaction` parameter and
53//! return `StmResult`.
54//!
55//! Use ? on `StmResult`, to propagate a transaction error through the system.
56//! Do not handle the error yourself.
57//!
58//! ```
59//! # use fast_stm::{atomically, TVar};
60//! let var = TVar::new(0);
61//!
62//! let x = atomically(|trans| {
63//!     var.write(trans, 42)?; // Pass failure to parent.
64//!     var.read(trans) // Return the value saved in var.
65//! });
66//!
67//! println!("var = {}", x);
68//! // var = 42
69//!
70//! ```
71//!
72//! # Transaction safety
73//!
74//! Software transactional memory is completely safe in the rust sense, so
75//! undefined behavior will never occur.
76//! Still there are multiple rules that
77//! you should obey when dealing with software transactional memory.
78//!
79//! * Don't run code with side effects, especially no IO-code.
80//!   Transactions repeat in failure cases. Using IO would repeat this IO-code.
81//!   Return a closure if you have to.
82//! * Don't handle `StmResult` yourself.
83//!   Use `Transaction::or` to combine alternative paths and `optionally` to check if an inner
84//!   function has failed. Always use `?` and
85//!   never ignore a `StmResult`.
86//! * Don't run `atomically` inside of another. `atomically` is designed to have side effects
87//!   and will therefore break transaction safety.
88//!   Nested calls are detected at runtime and handled with panicking.
89//!   When you use STM in the inner of a function, then
90//!   express it in the public interface, by taking `&mut Transaction` as parameter and
91//!   returning `StmResult<T>`. Callers can safely compose it into
92//!   larger blocks.
93//! * Don't mix locks and transactions. Your code will easily deadlock or slow
94//!   down unpredictably.
95//! * Don't use inner mutability to change the content of a `TVar`.
96//!
97//! Panicking in a transaction is transaction-safe. The transaction aborts and
98//! all changes are discarded. No poisoning or half written transactions happen.
99//!
100//! # Speed
101//!
102//! Generally keep your atomic blocks as small as possible, because
103//! the more time you spend, the more likely it is, to collide with
104//! other threads. For STM, reading `TVar`s is quite slow, because it
105//! needs to look them up in the log every time.
106//! Every used `TVar` increases the chance of collisions. Therefore you should
107//! keep the amount of accessed variables as low as needed.
108//!

134/// Convert a `TransactionClosureResult<T, E_A>` to `TransactionClosureResult<T, E_B>`.
135///
136/// This macro is used to cleanly write transactions where multiple kind of errors are
137/// possible during execution. The macro will not fail as long as the specified target
138/// error `$to` implements `From<E>`, `E` being the error possibly returned by `$op`.
139/// It expands to:
140///
141/// ```ignore
142/// $op.map_err(|e| match e {
143///         fast_stm::TransactionError::Abort(e) => fast_stm::TransactionError::Abort($to::from(e)),
144///         fast_stm::TransactionError::Stm(e) => fast_stm::TransactionError::Stm(e),
145///     })?
146/// ```
147///
148/// # Example
149///
150/// ```rust
151/// # use fast_stm::{abort, atomically_with_err, try_or_coerce, Transaction, TransactionClosureResult};
152///
153/// struct Error1;
154/// struct Error2;
155///
156/// impl From<Error1> for Error2 {
157///     fn from(e: Error1) -> Self {
158///         Error2
159///     }
160/// }
161///
162/// fn op1(trans: &mut Transaction) -> TransactionClosureResult<(), Error1> {
163///     // ...
164///     Ok(())
165/// }
166///
167/// fn op2(trans: &mut Transaction) -> TransactionClosureResult<(), Error2> {
168///     // ...
169///     Ok(())
170/// }
171///
172/// let res: Result<(), Error2> = atomically_with_err(|trans| {
173///     try_or_coerce!(op1(trans), Error2);
174///     op2(trans)?;   
175///     Ok(())
176/// });
177/// ```
178#[macro_export]
179macro_rules! try_or_coerce {
180    ($op: expr, $to: ident) => {
181        $op.map_err(|e| match e {
182            $crate::TransactionError::Abort(e) => $crate::TransactionError::Abort($to::from(e)),
183            $crate::TransactionError::Stm(e) => $crate::TransactionError::Stm(e),
184        })?
185    };
186}

189/// Call `abort` to abort a transaction and pass the error as the return value.
190///
191/// # Examples
192///
193/// ```
194/// # use fast_stm::*;
195/// struct MyError;
196///
197/// let execute_once: Result<u32, _> = atomically_with_err(|_| {
198///     abort(MyError)
199/// });
200///
201/// assert!(execute_once.is_err());
202/// ```
203pub fn abort<T, E>(e: E) -> TransactionClosureResult<T, E> {
204    Err(TransactionError::Abort(e))
205}

207#[inline]
208/// Call `retry` to abort an operation and run the whole transaction again.
209///
210/// Semantically `retry` allows spin-lock-like behavior, but the library
211/// blocks until one of the used `TVar`s has changed, to keep CPU-usage low.
212///
213/// `Transaction::or` allows to define alternatives. If the first function
214/// wants to retry, then the second one has a chance to run.
215///
216/// # Examples
217///
218/// ```no_run
219/// # use fast_stm::*;
220/// let infinite_retry: i32 = atomically(|_| retry());
221/// ```
222pub fn retry<T>() -> StmClosureResult<T> {
223    Err(StmError::Retry)
224}

226/// Run a function atomically by using Software Transactional Memory.
227/// It calls to `Transaction::with` internally, but is more explicit.
228pub fn atomically<T, F>(f: F) -> T
229where
230    F: Fn(&mut Transaction) -> StmClosureResult<T>,
231{
232    Transaction::with(f)
233}
234
235/// Run a function atomically by using Software Transactional Memory.
236/// It calls to `Transaction::with_err` internally, but is more explicit.
237pub fn atomically_with_err<T, E, F>(f: F) -> Result<T, E>
238where
239    F: Fn(&mut Transaction) -> TransactionClosureResult<T, E>,
240{
241    Transaction::with_err(f)
242}

244#[inline]
245/// Unwrap `Option` or call retry if it is `None`.
246///
247/// `optionally` is the inverse of `unwrap_or_retry`.
248///
249/// # Example
250///
251/// ```
252/// # use fast_stm::*;
253/// let x = TVar::new(Some(42));
254///
255/// atomically(|tx| {
256///         let inner = unwrap_or_retry(x.read(tx)?)?;
257///         assert_eq!(inner, 42); // inner is always 42.
258///         Ok(inner)
259///     }
260/// );
261/// ```
262pub fn unwrap_or_retry<T>(option: Option<T>) -> StmClosureResult<T> {
263    match option {
264        Some(x) => Ok(x),
265        None => retry(),
266    }
267}

269#[inline]
270/// Retry until `cond` is true.
271///
272/// # Example
273///
274/// ```
275/// # use fast_stm::*;
276/// let var = TVar::new(42);
277///
278/// let x = atomically(|tx| {
279///     let v = var.read(tx)?;
280///     guard(v==42)?;
281///     // v is now always 42.
282///     Ok(v)
283/// });
284/// assert_eq!(x, 42);
285/// ```
286pub fn guard(cond: bool) -> StmClosureResult<()> {
287    if cond {
288        Ok(())
289    } else {
290        retry()
291    }
292}

294#[inline]
295/// Optionally run a transaction `f`. If `f` fails with a `retry()`, it does
296/// not cancel the whole transaction, but returns `None`.
297///
298/// Note that `optionally` does not always recover the function, if
299/// inconsistencies where found.
300///
301/// `unwrap_or_retry` is the inverse of `optionally`.
302///
303/// # Example
304///
305/// ```
306/// # use fast_stm::*;
307/// let x:Option<i32> = atomically(|tx|
308///     optionally(tx, |_| retry()));
309/// assert_eq!(x, None);
310/// ```
311pub fn optionally<T, F>(tx: &mut Transaction, f: F) -> StmClosureResult<Option<T>>
312where
313    F: Fn(&mut Transaction) -> StmClosureResult<T>,
314{
315    tx.or(|t| f(t).map(Some), |_| Ok(None))
316}

367                    } else {

383    /// test if a STM calculation is rerun when a Var changes while executing
384    #[test]
385    fn read_write_interfere() {
386        use std::thread;
387        use std::time::Duration;
388
389        // create var
390        let var = TVar::new(0);
391        let varc = var.clone(); // Clone for other thread.
392
393        // spawn a thread
394        let t = thread::spawn(move || {
395            atomically(|tx| {
396                // read the var
397                let x = varc.read(tx)?;
398                // ensure that x varc changes in between
399                thread::sleep(Duration::from_millis(500));
400
401                // write back modified data this should only
402                // happen when the value has not changed
403                varc.write(tx, x + 10)
404            });
405        });
406
407        // ensure that the thread has started and already read the var
408        thread::sleep(Duration::from_millis(100));
409
410        // now change it
411        atomically(|tx| var.write(tx, 32));
412
413        // finish and compare
414        let _ = t.join();
415        assert_eq!(42, var.read_atomic());
416    }
